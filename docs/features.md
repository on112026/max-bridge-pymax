# Возможности и команды бота

Полный список команд Telegram-бота и ограничений протокола MAX,
которые нужно знать пользователю моста.

## Команды бота

### Базовые

| Команда | Что делает |
| --- | --- |
| `/start` | приветствие + клавиатура (`📥 Слушать MAX`, `📚 Чаты`, `ℹ️ Статус`, `🆘 Помощь`) |
| `/chats` | список MAX-чатов из локальной БД |
| `/status` | состояние MAX-клиента (`auth.status`, `last_error`) |
| `/help` | краткая справка |

### Авторизация и реплай

| Команда | Что делает |
| --- | --- |
| `/code 12345` | отдать SMS-код MAX (после `need_2fa`) |
| `/reauth_sms` | пересоздать MAX-клиент с чистого листа (стереть `/app/cache`) |
| `/reply <текст>` | ответить автору последнего входящего |
| `💬 Ответить` (inline) | то же с подстановкой текста под сообщением |

Подробнее про SMS/2FA flow — [`auth.md`](./auth.md).

### Супергруппа: топики = MAX-чаты

| Команда | Что делает |
| --- | --- |
| `/setgroup` | привязать TG-супергруппу как «панель управления» мостом |
| `/supergroup` | показать/сменить привязанную супергруппу |
| `/prune_topics` | синхронизировать топики: удалить устаревшие, переименовать по MAX-чатам |

Каждый MAX-чат, в котором участвует бот-аккаунт MAX, = один топик в
привязанной TG-супергруппе. Входящие из MAX приходят в нужный топик,
исходящие из Telegram уходят в нужный MAX-чат по `message_thread_id`.

### Chat-ops (admin-операции над MAX)

| Команда | Что делает |
| --- | --- |
| `/resolve <ссылка>` | превью чата перед вступлением (без join) |
| `/join <ссылка>` | вступить в группу/канал MAX по ссылке |
| `/invite <chat_id> <user_id\|+phone>` | пригласить пользователя |
| `/search_user <+79…>` | найти `user_id` по номеру телефона |
| `/pending <chat_id>` | список заявок на вступление |
| `/approve <chat_id> <user_id> [...]` | принять заявки |
| `/decline <chat_id> <user_id> [...]` | отклонить заявки |
| `/chatops` | список всех chat-ops команд |

Все chat-ops идут через очередь `chat_ops_queue` и выполняются в
MAX-процессе через `pymax.Client`. Синхронные операции
(`/search_user`, `/pending`) ждут результат через polling
`GET /chat_ops/{id}?wait=true`.

## Что мы шлём в Telegram

| В MAX | В Telegram |
| --- | --- |
| Текст | текст + Markdown-экранирование |
| Фото (одно) | `send_photo` + caption |
| Видео (одно, ≤ 49 МБ) | `send_video` + caption |
| Файл (≤ 49 МБ) | `send_document` + caption |
| Голосовое / стикер / аудио / видео-сообщение | текст «🎤 Голосовое сообщение (N сек)» + плейсхолдер |
| Несколько вложений | берём первое; остальные отбрасываем (Telegram API принимает одно) |

49 МБ — лимит Telegram Bot API для `sendDocument` / `sendVideo` при
аплоаде через локальный URL.

### Compact-режим для топиков supergroup (`COMPACT_TOPIC_MESSAGES`)

По умолчанию (`COMPACT_TOPIC_MESSAGES=false`) входящие из MAX в топиках
супергруппы пересылаются с шапкой `💬 чат / ↘️ автор · время` и тремя
inline-кнопками (`💬 Ответить` / `📋 ID чата` / `🔄 История`).

Если выставлено `COMPACT_TOPIC_MESSAGES=true` (см. `.env.example`),
`EventPoller` в `bot/app/forwarder.py` переключается в «мессенджерный»
режим для топиков:

* **Без шапки.** Топик уже назван по `chat_title` + тип чата, Telegram
  сам показывает автора/время у сообщения — дублировать лишнее.
* **Без inline-кнопок.** В топике автоэхо (`handlers/auth/topic_echo.py`)
  уже ловит любое сообщение и шлёт его в MAX; история чата лежит в том
  же топике сверху; `max_chat_id` не нужен (эхо знает топик само).
* **Реакция `📨` на входящее** — тихое подтверждение доставки (как `✅`
  на исходящее в `topic_echo.py`).

Compact-режим действует **только** на топики supergroup. В личке с ботом
и в `/history` сохраняется старый формат с шапкой и кнопками, потому
что там контекст «голый» и кнопки реально нужны.

## Что мы шлём в MAX

Из Telegram в MAX уходит текст и **ровно одно** вложение
(`Photo` / `Video` / `File` из PyMax), как описано в `/reply` бота.
Привязка «TG-сообщение → MAX-чат» — через `message_thread_id` топика.

## Реакции (лайки / смайлики)

Мост двусторонне синхронизирует **emoji-реакции** между MAX и Telegram
на сообщениях из MAX в привязанной супергруппе. Реализация использует
`pymax.Client.add_reaction` / `remove_reaction` и
`aiogram.Bot(SetMessageReaction)`.

### Направления

| Контекст в MAX | Что происходит в Telegram |
|---|---|
| **ЛС (DIALOG)** с владельцем | Зеркальная реакция бота (`setMessageReaction`) — бот ставит в TG ровно ту эмодзи, которую поставил оппонент в MAX (первую из `counters`). При снятии в MAX — бот снимает (`op=remove`). Без кнопок, без сводки. |
| **Группа / канал (CHAT, CHANNEL)** | Зеркальная реакция бота **+** inline-кнопка `🔄 Реакции` под сообщением **+** авто-карточка-сводка «👍×N 🔥×M · итого K» (отдельное сообщение, редактируется на каждый `on_reaction_update`). В CHAT/CHANNEL в TG зеркалится **только** своя реакция владельца (`your_reaction` через `get_reactions`); чужие реакции идут только в сводку. |

> **Поведение в DIALOG реализовано без state**: при первом изменении бот
> ставит в TG первую эмодзи из `counters`; при снятии (`counters=[]`) — снимает.
> Для DIALOG это эквивалентно «помнить какую эмодзи ставил бот», потому что
> оппонент в MAX может поставить только одну эмодзи на сообщение. Если в
> будущем MAX начнёт отдавать несколько разных эмодзи на одно сообщение в
> DIALOG — нужно будет добавить таблицу «какую эмодзи бот зеркалит для
> max_chat/max_msg» (см. `bridge._on_reaction_update`).

### Ограничения (документируем явно)

1. **Зеркалятся только реакции владельца моста** — того, чей MAX-аккаунт
   привязан (`MAX_PHONE`). Реакции остальных участников MAX-чата в TG
   **не показываются как настоящие реакции** — Bot API не позволяет
   боту ставить реакции «от имени других пользователей». В группе/канале
   они агрегируются в **текстовую сводку** «👍×3 🔥×1 · итого 4» под
   сообщением.
2. **Один emoji за раз** (стандарт MAX): владелец может поставить только
   одну свою реакцию; в TG бот ставит ровно одну ботовскую реакцию.
   При смене emoji — старая снимается, новая ставится (на стороне MAX
   `add_reaction` заменяет).
3. **Бот должен иметь право `setMessageReaction`** в супергруппе —
   т.е. быть админом с правом «Add reactions». Иначе вызовы молча
   падают (поведение совпадает с уже существующим `_mark_incoming_reaction`
   и `_echo_react`).
4. **Сводка** старше 48 часов не редактируется (лимит Telegram) — воркер
   `ReactionsMaxPoller` удаляет старую сводку и шлёт новую под тем же
   сообщением.
5. **`custom_emoji` и `paid` (Stars)** из TG не зеркалятся в MAX —
   MAX их не поддерживает; они просто игнорируются.
6. **API MAX отдаёт только счётчики по emoji**, без разбивки «кто
   именно поставил». Поэтому в сводке видно «👍 × 3», но не имена
   пользователей. Если MAX в будущем начнёт отдавать user_id —
   можно расширить.

### Архитектура (для разработчика)

* Очередь `reaction_ops_queue` (см. `shared/db/reaction_ops.py`) с тремя
  направлениями: `to_max` / `to_tg` / `to_tg_summary`. Та же таблица
  используется для трёх разных потоков, разделение — по колонке
  `direction`.
* MAX-процесс: `on_reaction_update` в `max/app/bridge/__init__.py`
  кладёт `to_tg` (your_reaction) и `to_tg_summary` (counters). Polling
  `reactions_loop` в `max/app/reactions_loop.py` обрабатывает только
  `to_max` (через `client.add_reaction` / `remove_reaction`) и `fetch_summary`
  (через `client.get_reactions`).
* Бот-процесс: `MessageReactionUpdated` в `bot/app/handlers/reactions_tg.py`
  кладёт `to_max` (add/remove). `ReactionsMaxPoller` в
  `bot/app/handlers/reactions_max.py` обрабатывает `to_tg` (`setMessageReaction`)
  и `to_tg_summary` (`editMessageText` / создание нового сообщения под
  исходным).
* Callback-кнопка `🔄 Реакции` (только в CHAT/CHANNEL) кладёт
  `to_max` с `op=fetch_summary` — MAX-процесс делает свежий
  `get_reactions` и кладёт `to_tg_summary`.
* Обратная TG-ссылка (`delivered_messages.tg_*`) заполняется в
  `bot/app/forwarder.py` через `POST /events/{id}/tg-mapping` сразу
  после успешной отправки сообщения.

## Ограничения PyMax 2.2.0

Из того, что **нельзя** сделать через мост на текущей версии PyMax:

- **Поиск пользователя по имени/нику** — только по телефону
  (`search_by_phone`). `/search_user` принимает только E.164.
- **Глобальный поиск публичных каналов/групп по подстроке** — не
  реализован в публичном протоколе. `/resolve` и `/join` работают
  только по полной ссылке `https://max.ru/join/<token>`.
- **Несколько вложений за раз** в обе стороны (см. таблицы выше).
- **Inline-кнопки в MAX** не форвардятся в Telegram.
- **Реакции остальных участников MAX-чата** не отражаются в TG как
  настоящие реакции (см. раздел «Реакции» выше) — только агрегируются
  в текстовую сводку.
- **Тред-реплай в MAX** не реализован.
